## ADR

# Architecture<br>Decision Record

新しいものではなく、昔からあるやり方です。<br>
Microsoft、AWS、Googleの公式ドキュメントでも紹介されています。

---

## ざっくり言うと

# 重要な設計判断を<br>決まった形で残す

---

## ADRに残すもの

- 判断したこと
- そのときの状況と理由
- 却下した代替案
- 影響やトレードオフ

---

## AIに渡したのは

# この2つだけ

- **どう残すか** — ADRのテンプレート  
  背景・決定・代替案・影響を、同じ形式で記録する
- **いつ残すか** — ADRを追加するルール  
  重要な設計変更があったときだけ、新しいADRを作る

---

## SKILLとテンプレートは、こんな感じ

```markdown
重要な設計変更がある場合はADRを追加する。

## Status
## Context
## Decision
## Alternatives
## Consequences
```

## たとえば、ADR-0017

```text
Context
  ContainersViewModelが約1,000行

Decision
  共有ファサードを残して内部を4分割

Rejected
  画面ごとにViewModelを分ける
  → 共有状態が壊れるため
```

## 設計書のほうは、こんな感じ

```markdown
> 現時点のスナップショットです。
> 経緯・検討過程は書きません。

ContainersViewModelは共有ファサード。
List / Details / Logs / Shellを所有する。
```

今の構成を書き、判断の理由はADRへリンクします。

---

## 別のリポジトリでは、決定を変更

```text
ADR-0002  BlobTriggerを採用
     ↓    Blob削除を検出できない
ADR-0003  Timer定期クロールへ変更
```

古いADRは上書きせず、`Superseded`として残します。
